## CATEGORY 2 — Realtime & Subscription Bugs

### 2.1 :white_check_mark: FIXED — Calendar realtime UPDATE keeps entries that moved to a different month

**File:** `app/(app)/calendar/CalendarClient.js:197`

**Bug:** The realtime UPDATE handler replaces an entry in-place:
```js
setEntries(prev => prev.map(e => e.id === payload.new.id ? payload.new : e))
```
When `markCalendarEntryDone` moves a calendar entry's `date` from the current viewed month to a different month (e.g., planned for April 15 but done on March 20), the UPDATE handler replaces the entry with one whose date is now outside the viewed month. This ghost entry appears on the wrong day of the current month grid.

**Reproduction:** View April on the calendar. A couple entry is on April 15. Mark it done with completion date March 20. The entry remains in April's state but now has date `2026-03-20`, so it doesn't appear on any cell (the grid only renders April dates). BUT the `entries.length` count in the subtitle is wrong, and if the entry's date happens to match a valid April day number, it shows up in the wrong day.

**User experience:** The entry count shows inflated numbers. The old dot on the original date disappears but the data structure is polluted.

**Fix:**
```js
// app/(app)/calendar/CalendarClient.js, line 197
} else if (payload.eventType === 'UPDATE') {
  const rowDate = payload.new?.date
  if (rowDate) {
    const [rY, rM] = rowDate.split('-').map(Number)
    if (rY !== viewYear || rM - 1 !== viewMonth) {
      // Entry moved out of current month — remove it
      setEntries(prev => prev.filter(e => e.id !== payload.new.id))
      return
    }
  }
  setEntries(prev => prev.map(e => e.id === payload.new.id ? payload.new : e))
}
```

**Resolution (2026-04-10):** Fixed in `app/(app)/calendar/CalendarClient.js` — the UPDATE handler now checks if the updated entry's date falls outside the currently viewed month. If it does, the entry is removed from state via `filter` instead of being updated in place.

## CATEGORY 3 — Server Action Bugs

### 3.1 :white_check_mark: FIXED — No guard against double-marking a bucket item as done (duplicate memories)

**File:** `app/actions/bucket.js:69-96`

**Bug:** `markAsDone` fetches the bucket item's name and category but does NOT check if `is_done` is already `true`. If called twice (double-click, race condition, or two partners marking simultaneously), it:
1. Sets `is_done: true` again (no-op, harmless)
2. Creates a SECOND memory record for the same bucket item
3. Moves the calendar entry date again (no-op if same date)

This creates duplicate memories in the database.

**Reproduction:** Two partners both tap "Mark done" on the same bucket item nearly simultaneously. OR: a slow network causes the user to tap "Mark done" again before the first request completes (the button is disabled by `isPending`, but the MarkDoneSheet is a separate form submission via `useActionState` which may not fully prevent resubmission).

**User experience:** The Memories page shows the same memory twice. If "Undo done" is used on one, the bucket item becomes undone but the other duplicate memory persists.

**Fix:**
```js
// app/actions/bucket.js, after line 77
if (!bucketItem) return { error: 'Item not found.' }

// ADD: Guard against double mark-done
const { data: existing } = await supabase
  .from('bucket_items')
  .select('is_done')
  .eq('id', bucketItemId)
  .eq('couple_id', profile.couple_id)
  .single()
if (existing?.is_done) return { error: 'This item is already marked as done.' }
```

Or more efficiently, fetch `is_done` in the existing select:
```js
const { data: bucketItem } = await supabase
  .from('bucket_items')
  .select('name, category, is_done')  // ADD is_done
  .eq('id', bucketItemId)
  .eq('couple_id', profile.couple_id)
  .single()

if (!bucketItem) return { error: 'Item not found.' }
if (bucketItem.is_done) return { error: 'This item is already marked as done.' }
```

Also add a similar guard in `markCalendarEntryDone` (`calendar.js:96-103`):
```js
// After fetching the entry
if (entry.bucket_item_id) {
  const { data: item } = await supabase
    .from('bucket_items')
    .select('is_done')
    .eq('id', entry.bucket_item_id)
    .single()
  if (item?.is_done) return { error: 'This entry has already been marked as done.' }
}
```

**Resolution (2026-04-07):** Fixed in `app/actions/bucket.js` — added `is_done` to the existing select in `markAsDone` and return early with an error if the item is already done. Also applied the same guard in `app/actions/calendar.js` inside `markCalendarEntryDone` for the linked bucket item check.

## CATEGORY 8 — Race Conditions & Async Bugs

### 8.1 :yellow_circle: MEDIUM — Bulk operations don't call `refetchItems()` / `refetch()` after server action

**Files:**
- `app/(app)/bucket/BucketClient.js:290-306` — `handleConfirmBulkDone`

- `app/(app)/memories/MemoriesClient.js:80-93` — `handleUndoDone`
- `app/(app)/memories/MemoriesClient.js:101-116` — `handleConfirmDelete`

**Bug:** These handlers use optimistic UI — they immediately update local state, then call the server action. On success, they rely on realtime events to keep state synced. **Reproduction:** Enable airplane mode briefly after submitting a bulk action. Realtime events are missed. The local state shows the optimistic update, but the server state is now different from what the UI shows (if the action failed silently or partially).

**User experience:** State drift between what the user sees and what's in the database. The next mount refetch corrects it, but until then, actions on phantom items could fail.

**Fix:** Add a refetch call after successful bulk operations:
```js
// In handleConfirmBulkDone, after line 300
const result = await bulkMarkDone(ids, date)
if (result?.error) {
  // ... rollback ...
} else {
  await refetchItems()  // ADD
}
```

### 9.1 :white_check_mark: FIXED — ProfileClient `useEffect` uses stale `draft` value

**File:** `app/(app)/profile/ProfileClient.js:14-19`

**Bug:**

```js
useEffect(() => {
  if (state?.success) {
    // Use the value that was actually submitted via formData,
    // not the current draft which may have changed
    setDisplayName(prev => draft)  // Or better: read from state
    setEditing(false)

}, [state, draft])
```

Or better, return the saved name from the server action:
```js
// In app/actions/profile.js
return { success: true, name }  // Return the saved name

// In ProfileClient.js
useEffect(() => {
  if (state?.success) {
    setDisplayName(state.name)
    setDraft(state.name)
    setEditing(false)
  }
}, [state])
```

**Resolution (2026-04-10):** Fixed in `app/actions/profile.js` and `app/(app)/profile/ProfileClient.js` — the server action now returns `{ success: true, name }` with the saved name, and `ProfileClient` reads `state.name` directly instead of the possibly-stale `draft` value.

### 9.2 :yellow_circle: MEDIUM — Multiple `useEffect` hooks missing `onSuccess` dependency

**Files:**
- `app/(app)/bucket/AddBucketForm.js:23-25`
- `app/(app)/bucket/MarkDoneSheet.js:26-28`
- `app/(app)/calendar/AddCalendarEntryForm.js:19-21`
- `app/(app)/calendar/AddMemoryForm.js:18-20`
- `app/(app)/calendar/CalendarMarkDoneSheet.js:26-28`
- `app/(app)/memories/AddMemoryForm.js:19-21`

**Bug:** All these components have:
```js
useEffect(() => {
  if (state?.success) onSuccess()
}, [state])  // missing onSuccess
```

React's rules of hooks require all values used inside an effect to be listed in the dependency array. The `onSuccess` callback is typically an inline function that changes reference each render. While this works in practice (the effect only fires when `state` changes), it violates React's exhaustive-deps rule and could lead to calling a stale `onSuccess` in edge cases.

**User experience:** In practice, this rarely causes issues because `state?.success` only becomes true once. But if `onSuccess` were to change between the form submission and the response (e.g., due to parent re-render), the old callback would be invoked.

**Fix:** Add `onSuccess` to the dependency array in all affected files:
```js
useEffect(() => {
  if (state?.success) onSuccess()
}, [state, onSuccess])
```

And wrap `onSuccess` callbacks in `useCallback` at the call site if needed to prevent infinite loops.

### 10.3 :green_circle: LOW — Invite code generation has no collision retry

**File:** `app/actions/couple.js:8-13`

**Bug:** `generateInviteCode()` generates a 6-character alphanumeric code (31^6 = ~887M combinations). The `invite_code` column has a unique constraint, so a collision would cause a Supabase insert error at line 27-30, returning a generic "Could not create couple space" error.

**Reproduction:** Extremely unlikely with a small user base. Would require ~30,000 couples before collision probability reaches 0.05% (birthday paradox).

**User experience:** On the rare collision, user sees "Could not create couple space. Please try again." Retrying generates a new code which almost certainly won't collide.

**Fix:** Add a retry loop (optional, given the low probability):
```js
let couple, coupleError
for (let attempt = 0; attempt < 3; attempt++) {
  const inviteCode = generateInviteCode()
  const result = await supabase.from('couples')
    .insert({ invite_code: inviteCode, creator_id: user.id })
    .select('id, invite_code').single()
  if (!result.error) { couple = result.data; break }
  coupleError = result.error
}
if (!couple) return { error: 'Could not create couple space. Please try again.' }
```
